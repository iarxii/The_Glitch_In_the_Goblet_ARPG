# Performance Telemetry Report
**Date:** 2026-03-06

## Methodology
An automated headless browser agent was utilized to navigate to the local development server (`http://localhost:5173`). The agent waited for 10 seconds to allow the initial Voxel generation and Rapier physics instantiation to resolve. It then scraped the live React DOM for the hidden `#agent-telemetry-data` node injected by the `<TelemetryProvider>`.

## Raw Data Dump
```json
{
  "fps": 5,
  "renderTimeMs": 192.4,
  "drops": 11
}
```

## Analysis & Findings

1.  **Critical Framerate Drop:** The application is running at a virtually unplayable **5 FPS**.
2.  **Render Pipeline Bottleneck:** The React `<Profiler>` reported an average `actualDuration` of **192.4ms** per frame for the `<GameRoot>`. In a 60 FPS application, a frame must render in under 16.6ms. We are exceeding the rendering budget by over 1000%.
3.  **Frame Drops:** 11 severe frame drops (delta > 0.1s) occurred during the 10-second observation window, likely corresponding to garbage collection spikes or physics step freezing.

## Conclusion
Despite the implementation of visual `InstancedMesh` rendering and 1D Greedy Meshing for the Rapier colliders, the React reconciliation and/or Three.js render cycle is completely bottlenecked. 

### Recommended Next Steps for Investigation:
1.  **React Re-renders:** It's highly probable that the `World.tsx` or `Chunk.tsx` components are rapidly re-rendering every frame due to unstable Zustand state subscriptions (e.g., subscribing to the player's rapidly changing position).
2.  **Physics Debugging:** If the physics step is taking too long despite greedy meshing, we may need to investigate if Rapier's `InstancedRigidBodies` vs `RigidBody` reconciliation is thrashing memory.
3.  **Shadow Map Resolution:** The massive chunks casting shadows on themselves may be exceeding the GPU's shadow-map drawing capacity. Disabling `castShadow` and `receiveShadow` on the `InstancedMesh` as a test could isolate GPU fill-rate vs CPU React bound issues.
